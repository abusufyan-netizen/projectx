🔍 Introduction & Core Scope

Introduction: The Academic Resource & Research Hub is a dynamic knowledge portal designed for faculty and students. It transforms static paper repositories into a collaborative research ecosystem. Faculty publish versioned research papers, students can “fork” resources for derivative work, and nested discussion threads encourage academic peer feedback. Built with HTML, CSS, JS, PHP – pure web engineering stack.

✅ In-Scope (Core deliverables)

  • Document Versioning (manual v1, v2 via file renaming)
  • Forking mechanism – copy paper to student workspace + attribution
  • Categorization Engine: search/filter by subject, year, difficulty
  • Discussion Thread: nested comments (reply-to-reply) per resource
  • Soft‑Delete system + data recovery (recycle bin)
  • Robust file validation: MIME type, extension, size before server save
  • Multi-level permissions: Student (fork/comment) vs Faculty (upload/delete)
  • File‑path mapping & relative storage architecture

❌ Out-of-Scope (Clarity control)

  • No real‑time notifications / WebSockets / email alerts
  • No advanced AI or plagiarism detection
  • No SSO / OAuth – session‑based login only
  • No Docker/cloud deployment (local XAMPP / WAMP)
  • No automated background jobs (cron not mandatory)

Focus remains on PHP business logic, database integrity, and interactive UI.

⚙️ Engineering "Wow" Factor Implementation

✨ Soft‑Delete System + Data Recovery: Instead of permanent deletion, papers are moved to recycle_bin table with timestamp and restore_key. Faculty can restore within 7 days (configurable). Actual files remain in /recycle directory until purged.

✨ File Validation Module (Pre‑server validation): Using PHP's finfo_file() to detect true MIME type, checking extension whitelist and size limit. Invalid uploads are rejected instantly, never saved to server. Below is the core logic pattern:

// File validation snippet (PHP) – prevents malicious uploads
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$mime = finfo_file($finfo, $_FILES['paper']['tmp_name']);
$allowed_mimes = ['application/pdf', 'application/msword', 'application/vnd.openxmlformats-officedocument.wordprocessingml.document'];
$max_size = 10 * 1024 * 1024; // 10MB
if (!in_array($mime, $allowed_mimes)) die("Invalid MIME type.");
if ($_FILES['paper']['size'] > $max_size) die("File too large.");
// extension validation & safe filename generation

📁 Database Complexity Highlights: Nested comments stored using adjacency list (parent_id), recursive query rendering; Multi-level permissions (student, faculty, optional admin). File‑path mapping ensures uploaded files are stored with UUID names and relative path stored in `versions` table for portability.

🎯 Focus on Scope – Detailed Rationale

The primary objective is to deliver a fully functional web portal that demonstrates practical research collaboration. The team will concentrate on:

  • Document Versioning & Forking: Unique academic value – students can fork papers, build upon them, and faculty can track derivative works.
  • Nested Comment Threads: Mimics real academic discussion (Q&A style). Each resource page will support unlimited nested replies.
  • Security-First File Handling: Implementation of server-side validation prevents dangerous file uploads, a common web vulnerability.
  • Soft-Delete with Restore: Emphasizes data resilience and recovery – adds professional-grade feature.
  • Permission Enforcement: Faculty can edit/delete own papers, while students are limited to forking/commenting.

Tech Stack Constraints: Only HTML, CSS, JavaScript, PHP (vanilla) and MySQL. No frameworks like Laravel or React – ensures core web engineering understanding. The categorization engine will leverage SQL queries + AJAX for smooth user experience.

🗄️ Database Schema Blueprint (Core Tables)

📌 users(id, name, email, password, role: 'student'/'faculty', created_at)
📌 papers(id, title, abstract, subject, year, difficulty_level, author_id, parent_paper_id NULL, forked_from_id NULL, status 'active'/'deleted')
📌 versions(id, paper_id, version_number, file_path, uploaded_at, size)
📌 forks(id, original_paper_id, forked_by_user_id, forked_version_id, created_at)
📌 comments(id, paper_id, user_id, parent_comment_id NULL, content, created_at)
📌 recycle_bin(id, original_table, original_id, file_path_backup, deleted_by, deleted_at, restore_key)

Nested comment retrieval: using recursive PHP function or MySQL CTE (if MySQL 8+). Path mapping ensures uploaded files are stored under /uploads/{year}/{subject_hash}/ for organized retrieval.
